# Litho 边界接口

## 接口总览

Litho 是一个 CLI 工具，它的对外接口主要有三类：命令行接口（用户通过终端与 Litho 交互）、配置接口（用户通过 litho.toml 文件定义运行参数）、LLM Provider 接口（系统通过 HTTPS API 与 LLM 服务交互）。本章逐一详述这些接口的完整规格。

## CLI 接口

### 主命令：文档生成

这是 Litho 的默认命令——直接执行 `deepwiki-rs` 即可为当前项目生成文档。

```
deepwiki-rs [OPTIONS]
```

| 参数 | 类型 | 是否必须 | 说明 | 默认值 |
|------|------|---------|------|-------|
| `-p, --project-path` | PathBuf | 否 | 要分析的项目目录 | `.` (当前目录) |
| `-o, --output-path` | PathBuf | 否 | 文档输出目录 | `./litho.docs` |
| `-c, --config` | PathBuf | 否 | 配置文件路径 | 自动查找 `litho.toml` |
| `-n, --name` | String | 否 | 项目名称 | 自动推断（从 Cargo.toml/package.json 等） |
| `--skip-preprocessing` | bool | 否 | 跳过预处理阶段 | false |
| `--skip-research` | bool | 否 | 跳过研究阶段 | false |
| `--skip-documentation` | bool | 否 | 跳过文档编排阶段 | false |
| `-v, --verbose` | bool | 否 | 启用详细日志 | false |
| `--model-efficient` | String | 否 | 高效模型名称（用于常规推理） | 配置文件中的值 |
| `--model-powerful` | String | 否 | 强力模型名称（用于复杂推理） | 等于 model-efficient |
| `--llm-api-base-url` | String | 否 | LLM API 基础 URL | Provider 默认值 |
| `--llm-api-key` | String | 否 | LLM API 密钥 | 配置文件中的值 |
| `--max-tokens` | u32 | 否 | 最大 token 数 | 4000 |
| `--temperature` | f64 | 否 | LLM 温度参数 | 0.7 |
| `--max-parallels` | usize | 否 | 最大并行 LLM 调用数 | 3 |
| `--tool-concurrency` | usize | 否 | ReAct 工具调用并发数 | 5 |
| `--llm-provider` | String | 否 | LLM Provider 类型 | openai |
| `--target-language` | String | 否 | 文档输出语言 | en |
| `--disable-preset-tools` | bool | 否 | 禁用 ReAct 预设工具 | false |
| `--no-cache` | bool | 否 | 禁用缓存 | false |
| `--force-regenerate` | bool | 否 | 强制重新生成（清除缓存） | false |
| `--boundary-code-limit` | usize | 否 | 边界分析的代码洞察限制 | 50 |
| `--boundary-include-source` | bool | 否 | 边界分析中包含源代码 | true |
| `--boundary-only-directories-when-files-more-than` | usize | 否 | 文件数超过此阈值时只显示目录 | None |

**支持的 LLM Provider**：`openai`, `moonshot`, `deepseek`, `mistral`, `openrouter`, `anthropic`, `gemini`, `ollama`

**支持的目标语言**：`zh`, `en`, `ja`, `ko`, `de`, `fr`, `ru`, `vi`

**使用示例**：

```bash
# 为当前项目生成文档（默认设置）
deepwiki-rs

# 指定项目目录与输出目录
deepwiki-rs -p ./my-project -o ./docs

# 生成日语文档
deepwiki-rs --target-language ja -p ./my-project

# 使用 DeepSeek Provider 和自定义模型
deepwiki-rs --llm-provider deepseek --llm-api-base-url https://api.deepseek.com --llm-api-key <key> --model-efficient deepseek-chat

# 跳过预处理和研究（仅重新编排文档）
deepwiki-rs --skip-preprocessing --skip-research

# 禁用 ReAct 工具调用
deepwiki-rs --disable-preset-tools --llm-api-base-url <base> --llm-api-key <key>
```

### 子命令：知识同步

用于预同步外部知识源（PDF/MD/SQL 文件），确保缓存是最新的。

```
deepwiki-rs sync-knowledge [OPTIONS]
```

| 参数 | 类型 | 是否必须 | 说明 | 默认值 |
|------|------|---------|------|-------|
| `-c, --config` | PathBuf | 否 | 配置文件路径 | 自动查找 `litho.toml` |
| `--force` | bool | 否 | 强制同步（即使缓存未过期） | false |

**使用示例**：

```bash
# 同步外部知识源（处理并缓存本地文档）
deepwiki-rs sync-knowledge

# 强制重新同步
deepwiki-rs sync-knowledge --force

# 使用特定配置文件
deepwiki-rs sync-knowledge -c ./my-litho.toml
```

## 配置接口 (litho.toml)

Litho 的配置文件使用 TOML 格式，支持以下配置结构。完整的配置示例参见 `litho-example.toml`。

### 核心配置项

| 配置项 | 类型 | 描述 | 默认值 |
|-------|------|------|-------|
| `project_name` | String | 项目名称（覆盖自动推断） | 自动从 Cargo.toml/package.json 推断 |
| `project_path` | PathBuf | 要分析的项目目录 | `.` |
| `output_path` | PathBuf | 文档输出目录 | `./litho.docs` |
| `target_language` | String | 文档输出语言 | `en` |
| `analyze_dependencies` | bool | 是否分析代码依赖关系 | true |
| `identify_components` | bool | 是否识别组件 | true |
| `max_depth` | u8 | 目录扫描最大深度 | 10 |
| `max_file_size` | u64 | 单文件最大大小（字节） | 1048576 |
| `include_tests` | bool | 是否包含测试文件 | false |
| `include_hidden` | bool | 是否包含隐藏文件 | false |
| `git_tracked_only` | bool | 是否只分析 Git 追踪的文件 | false |
| `excluded_dirs` | Vec<String> | 排除的目录列表 | [] |
| `excluded_files` | Vec<String> | 排除的文件列表 | [] |
| `excluded_extensions` | Vec<String> | 排除的文件扩展名 | [] |
| `included_extensions` | Vec<String> | 包含的文件扩展名 | [] |

### LLM 配置项

| 配置项 | 类型 | 描述 | 默认值 |
|-------|------|------|-------|
| `llm.provider` | String | LLM Provider 类型 | `openai` |
| `llm.api_key` | String | API 密钥 | "" |
| `llm.api_base_url` | String | API 基础 URL | Provider 默认值 |
| `llm.model_efficient` | String | 高效模型名称 | `gpt-4o-mini` |
| `llm.model_powerful` | String | 强力模型名称 | 等于 model_efficient |
| `llm.max_tokens` | u32 | 最大 token 数 | 4000 |
| `llm.temperature` | f64 | 温度参数 | 0.7 |
| `llm.retry_attempts` | u32 | 重试次数 | 3 |
| `llm.retry_delay_ms` | u64 | 重试延迟（毫秒） | 1000 |
| `llm.timeout_seconds` | u64 | 调用超时（秒） | 120 |
| `llm.disable_preset_tools` | bool | 禁用预设工具 | false |
| `llm.max_parallels` | usize | 最大并行数 | 3 |
| `llm.max_turns` | usize | ReAct 最大循环次数 | 30 |
| `llm.tool_concurrency` | usize | 工具并发数 | 5 |

### 缓存配置项

| 配置项 | 类型 | 描述 | 默认值 |

### 配置文件示例

```toml
project_name = "my-project"
project_path = "."
output_path = "./docs"
target_language = "zh"

[llm]
provider = "openai"
api_key = "your-api-key"
api_base_url = "https://api.openai.com/v1"
model_efficient = "gpt-4o-mini"
model_powerful = "gpt-4o"
max_tokens = 4000
temperature = 0.7

[cache]
enabled = true
expire_hours = 168

[knowledge.local_docs]
enabled = true
watch_for_changes = true

[[knowledge.local_docs.categories]]
name = "architecture"
description = "系统架构文档"
paths = ["docs/architecture/**/*.md"]
target_agents = ["SystemContextResearcher", "ArchitectureResearcher"]
```

## LLM Provider 接口

Litho 与 LLM Provider 的交互全部通过 HTTPS API 完成。以下是各 Provider 的通信规格：

| Provider | API 协议 | 默认 Base URL | 结构化输出支持 | 工具调用支持 |
|---------|---------|-------------|------------|------------|
| OpenAI | HTTPS POST | `https://api.openai.com/v1` | ✅ (JSON Schema) | ✅ (Function Calling) |
| Moonshot | HTTPS POST | `https://api.moonshot.cn/v1` | ✅ | ✅ |
| DeepSeek | HTTPS POST | `https://api.deepseek.com` | ✅ | ✅ |
| Mistral | HTTPS POST | `https://api.mistral.ai` | ✅ | ✅ |
| OpenRouter | HTTPS POST | `https://openrouter.ai/api/v1` | ✅ | ✅ |
| Anthropic | HTTPS POST | `https://api.anthropic.com` | ✅ | ✅ (Tool Use) |
| Gemini | HTTPS POST | `https://generativelanguage.googleapis.com` | ✅ | ✅ |
| Ollama | HTTP POST | `http://localhost:11434` | ✅ (自定义) | ✅ |

**请求格式**：所有 Provider 都使用 rig-core 的统一接口发送请求，内部由 ProviderClient enum 路由到对应的 SDK 实现。

**响应格式**：
- Extract 模式：返回 JSON 格式的结构化数据
- Prompt 模式：返回纯文本响应
- ReAct 模式：返回多轮交互结果（推理步骤 + 工具调用 + 最终回答）

**回退机制**：OpenAI 和 Ollama 的结构化输出调用失败时，回退到直接 HTTP POST 请求（由 OpenAICompatibleExtractorWrapper 和 OllamaExtractorWrapper 处理）。

## LLM 工具接口（ReAct 模式）

在 ReAct 推理模式下，Agent 可以使用以下工具自主探索项目：

| 工具名称 | 路径 | 功能 | 参数 |
|---------|------|------|------|
| `FileExplorer` | `src/llm/tools/file_explorer.rs` | 列出目录内容、搜索文件 | `directory_path`, `pattern` |
| `FileReader` | `src/llm/tools/file_reader.rs` | 读取文件内容 | `file_path`, `max_lines` |
| `Time` | `src/llm/tools/time.rs` | 获取当前时间 | 无 |

这些工具通过 rig-core 的 Tool trait 注册到 Agent，LLM 在 ReAct 循环中根据推理需要自主决定是否调用。

## 集成建议

### CI/CD 集成

```bash
# 在 CI 脚本中集成 Litho
deepwiki-rs -p ./src --llm-api-key $LLM_API_KEY --model-efficient gpt-4o-mini --no-cache
```

### Ollama 本地模型集成

```bash
# 使用本地 Ollama 模型（无需 API Key）
deepwiki-rs --llm-provider ollama --model-efficient llama3
```

### 增量更新策略

```bash
# 利用缓存进行增量更新（只分析变更部分）
deepwiki-rs -p ./src  # 默认启用缓存，自动跳过未变更的分析

# 强制完全重新生成
deepwiki-rs -p ./src --force-regenerate
```
