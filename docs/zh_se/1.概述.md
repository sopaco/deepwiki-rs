# Litho (deepwiki-rs) 项目概述

## 系统定义

Litho 是一个用 Rust 构建的高性能 AI 文档生成引擎，它的使命很简单但极具价值——**让软件项目的架构文档永远与代码保持同步**。在传统的开发实践中，架构文档往往是最容易被忽视的角落：项目初期可能有人用心撰写过一份详尽的设计文档，但随着代码的不断迭代，这些文档很快就变得过时、残缺甚至完全失联。Litho 的出现就是为了终结这个困境——它像一个不知疲倦的架构师助理，时刻紧跟代码的变化，自动把最新的项目结构、模块职责、依赖关系和核心流程整理成专业的 C4 架构文档。

Litho 采用"静态代码分析 + AI 多智能体"的双引擎策略。先通过静态分析把源码的结构信息提取出来，再借助多个专业 AI 智能体从不同角度对项目进行深度理解——系统上下文、领域模块、架构模式、工作流程、边界接口逐一剖析，最终将所有研究成果编排成清晰可读的技术文档。这套流水线的设计哲学是：**静态分析提供精确的事实基础，AI 推理提供洞察和叙述，两者缺一不可**。

## 核心功能

| 功能 | 描述 | 解决的问题 |
|------|------|-----------|
| **C4 模型文档自动生成** | 自动生成 Context、Container、Component 三层 C4 架构图与配套文档 | 手工绘制 C4 图费时费力且极易过时 |
| **多语言代码分析** | 支持 Rust、Python、Java、Go、C#、JS/TS、Kotlin、PHP、Swift、React、Vue、Svelte 等 14 种语言 | 单语言分析器无法覆盖多语言项目 |
| **8 种 LLM Provider 适配** | 支持 OpenAI、Moonshot、DeepSeek、Mistral、OpenRouter、Anthropic、Gemini、Ollama | 用户可能使用不同的 LLM 服务 |
| **外部知识挂载** | 支持挂载 PDF、Markdown、SQL 等外部文档作为知识源 | 项目内部的代码信息不足以完整描述架构决策 |
| **数据库文档生成** | 自动分析 SQL 项目并生成包含 ER 图的数据库架构文档 | 数据库结构文档往往缺失或不完整 |
| **ReAct 推理模式** | Agent 通过"推理-行动"循环使用工具自主探索代码 | 单次 LLM 调用无法深入理解复杂项目 |
| **智能缓存** | 基于 MD5 哈希的持久化缓存，避免重复的 LLM 调用 | LLM API 调用成本高昂且耗时 |
| **Mermaid 图表自动修复** | 验证并修复生成文档中的 Mermaid 语法错误 | Mermaid 语法复杂，AI 生成的图常有语法问题 |

## 用户角色

用户角色这回事其实很简单——Litho 是一个 CLI 工具，不是 Web 服务，所以不存在"在线用户"的概念。它的用户都是坐在终端前面的开发者。

| 角色 | 描述 | 主要使用场景 |
|------|------|------------|
| **项目开发者** | 项目的核心开发者或新加入的团队成员 | 日常开发中随时生成最新的项目文档，帮助自己或同事理解项目架构 |
| **技术架构师** | 负责项目架构决策的技术负责人 | 定期审视架构文档，评估模块划分是否合理、依赖关系是否健康 |
| **DevOps 工程师** | 负责 CI/CD 流程和自动化部署的运维人员 | 在 CI 流程中集成 Litho，每次代码提交自动更新文档 |
| **开源项目维护者** | 维护开源项目需要持续更新文档的社区贡献者 | 为开源项目的 README 和 Wiki 提供自动化文档支持 |

## 外部系统依赖

Litho 的外部依赖比较精简，核心的交互几乎全部集中在 LLM API 调用和文件系统操作上。这种设计是有意为之的——作为 CLI 工具，它需要尽量减少对外部服务的依赖，保证在大多数环境下都能顺利运行。

| 外部系统 | 交互类型 | 数据流向 | 关键性 |
|---------|---------|---------|-------|
| **LLM Provider API** (OpenAI/Moonshot/DeepSeek 等) | HTTPS API 调用 | 出：Prompt 请求 → 入：LLM 响应 | 高 — 核心推理能力依赖此服务 |
| **Ollama 本地服务** | HTTP API 调用 | 出：Prompt 请求 → 入：模型响应 | 中 — 可选的本地 LLM 替代方案 |
| **文件系统** | 本地文件读写 | 双向：读取源码 + 写入文档 | 高 — 输入源和输出目标 |
| **Mermaid Fixer** (外部二进制) | 命令行调用 | 出：Mermaid 代码 → 入：修复结果 | 中 — 文档质量保障 |
| **PDF 文件** (外部知识) | 文件读取 | 入：PDF 内容作为知识源 | 低 — 可选的外部知识增强 |
| **Git 仓库** | 文件系统读取 | 入：追踪文件列表 | 中 — git_tracked_only 模式下使用 |

## 业务价值

- **解决的核心问题**：软件项目架构文档的手工编写和维护极其耗时，且文档极易与代码脱节。Litho 把这个过程完全自动化，让文档永远反映代码的真实状态。
- **不使用 Litho 时的替代方案**：开发者需要手动阅读源码、绘制架构图、编写模块说明，这些工作可能需要数天时间，而且每次代码变更后都要重复这个过程。
- **主要输入**：一个软件项目的源代码目录（支持多种语言）
- **主要输出**：一套完整的 C4 模型架构文档，包括项目概述、架构图、工作流描述、模块深度研究、边界接口清单、数据库概览（条件触发）

## 系统边界

**范围内**：
- 自动分析代码库结构并生成架构文档
- 多种编程语言的静态代码分析
- C4 模型（Context / Container / Component）文档与 Mermaid 图表
- 外部知识源挂载增强分析能力
- 数据库项目（SQL）的架构文档生成
- Mermaid 图表语法验证与修复
- 多种 LLM Provider 的适配与集成
- 缓存机制以减少重复的 LLM 调用

**范围外**：
- 不提供在线文档浏览服务（由 Litho Book 生态工具负责）
- 不修改源代码或影响项目编译
- 不执行代码运行或测试
- 不管理文档版本历史（依赖 Git 等外部工具）
- 不提供实时文档更新（需要手动或 CI 触发）

## 技术栈概览

| 层次 | 技术 | 选择理由 |
|------|------|---------|
| 编程语言 | Rust | 高性能、内存安全、零成本抽象——处理大型代码库时性能优势明显 |
| 异步运行时 | Tokio | Rust 生态最成熟的异步框架，支持并发 LLM 调用 |
| LLM 框架 | rig-core | 提供 Agent/Extractor/Tool 的统一抽象，简化多 Provider 适配 |
| CLI 解析 | clap (derive) | Rust 生态标准的 CLI 框架，声明式定义更清晰 |
| HTTP 客户端 | reqwest | 支持 async 和 JSON，适合与 LLM API 交互 |
| 序列化 | serde/serde_json | Rust 数据序列化的标准方案，与 TOML/JSON 配置无缝配合 |
| 错误处理 | anyhow/thiserror | anyhow 用于应用级错误链，thiserror 用于自定义错误类型 |
| 缓存策略 | 自研 CacheManager | 基于 MD5 哈希的文件缓存，兼顾持久性和缓存命中率监控 |
| 配置管理 | toml | 人类可读的配置格式，适合 CLI 工具的配置文件 |

## 项目定位与生态

Litho 并不是孤立存在的——它属于一个完整的文档生态体系。**Litho Book** 是高性能的 Markdown 阅读器（Rust + Axum），负责把 Litho 生成的文档以优雅的方式呈现给读者；**Mermaid Fixer** 是专门的 Mermaid 图表修复工具，确保生成的架构图语法无误。三者配合形成"生成 → 修复 → 展示"的完整闭环，而 Litho 在其中承担的是最核心的"生成"角色。
